# Build Standards

Use these standards after the marketer approves the wireframe.

## Local HTML Output

Create one local `.html` file by default.

The file should be:

- editable in a local code agent or editor
- self-contained unless the customer requests a multi-file app
- ready for browser QA
- ready to hand to the Folloze publish path after QA

If a durable file is required and there is no active workspace, ask where to create it.

## Page Strategy

The page should have one clear campaign argument.

Before writing visible copy, define:

- the audience's current situation
- the campaign tension or opportunity
- the promised value
- the proof available
- the action the page should make easier

If the argument is generic after swapping in another audience, account, or segment, sharpen it before building.

## Visual System

Use the customer's approved source for design direction:

- source website
- landing page
- brand guide
- screenshot
- design system
- asset pack

Capture:

- colors and neutral surfaces
- typography role and scale
- button variants
- card shape and spacing
- hero style
- proof and resource treatment
- dark and light section rhythm
- footer and final CTA style

Do not copy a public page pixel-for-pixel. Adapt the visual system to the approved wireframe.

## Copy

Write for the buyer or visitor, not for the production team.

Prefer:

- concrete outcomes
- campaign-specific nouns and verbs
- proof with implication
- CTA labels that describe the value of clicking
- short section copy that advances the story

Avoid buyer-facing text such as:

- generated by
- placeholder
- template
- internal brief
- wireframe
- draft
- QA
- this skill
- test page

Do not expose private notes, account scoring, contact behavior, internal meeting comments, sales-stage details, or raw intent data unless the marketer explicitly approves that visible use.

## Controls And Links

Every visible control must do real work:

- nav item
- CTA button
- resource link
- card action
- modal opener
- tab
- slider
- filter
- calculator
- form
- carousel control
- video or media control

No placeholder `href="#"`, `javascript:void(0)`, empty form action, unlabeled icon-only control, decorative arrow, or fake interaction should remain in the build.

External links should open safely with `target="_blank" rel="noopener"` unless the customer's publishing guide says otherwise.

## Analytics

When the Folloze publish path supports analytics hooks:

- track primary CTA clicks
- track secondary CTA clicks
- track resource clicks
- track internal anchor navigation
- track meaningful custom interactions such as tabs, filters, scenario selectors, calculator changes, modal opens, and form starts

Use clear payloads that include interaction text, page area, destination URL when applicable, and selected option when applicable.

## Responsive Behavior

Design for desktop and mobile from the start.

Requirements:

- no horizontal overflow on mobile
- nav and CTA labels fit without awkward wrapping
- hero content remains readable on short viewports
- cards and proof sections stack cleanly
- modals and drawers fit mobile screens
- large headings wrap safely
- tap targets are usable
- motion has reduced-motion fallback

## Asset Rules

Use approved or source-owned assets when possible.

Before QA:

- verify logos render on their actual backgrounds
- verify images load
- verify image crops and aspect ratios
- verify SVG fills and transparency
- replace broken, blocked, or misleading assets

Do not invent logos, customers, metrics, awards, case studies, or analyst proof to fill a design slot.
