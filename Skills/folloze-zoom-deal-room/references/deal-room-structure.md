# Deal Room Structure

Use this structure for the first version of a Folloze-owned deal room created from Zoom notes.

## MVP Sections

1. **Hero**
   - Account or audience context.
   - Outcome-focused headline.
   - Primary CTA for the next meeting or proposal review.
   - Secondary CTA for resource review.

2. **Decision Snapshot**
   - Three to five cards covering current situation, desired outcome, proof needed, and timeline.
   - Keep private details internal. Visible cards should describe evaluation needs, not transcript notes.

3. **Recommended Path**
   - A short sequence showing how Folloze helps the buyer move from current workflow to measurable launch.
   - For Folloze assets, align to Build, Activate, Signal when it fits the account context.

4. **Resource Center**
   - Examples, proposal, security/MSA, integration notes, customer proof, or event examples.
   - Every card must have a real link or open an in-page modal/drawer with useful content.

5. **Workflow Fit**
   - Show how Folloze fits around current tools such as event platforms, CRM, MAP, CMS, or AI content workflows.
   - For event-led deals, show pre-event page, live-event experience, and post-event follow-up.

6. **Mutual Action Plan**
   - Buyer tasks.
   - Folloze tasks.
   - Shared next meeting.
   - Decision or launch milestone.

7. **Team And Next Step**
   - Customer-facing Folloze owner cards when useful.
   - Final CTA for the next meeting, proposal review, or resource review.

## Section Variants

Use these variants when the Zoom recap calls for them:

- **Event Follow-Up Room**: event workflow, Luma or current tool fit, post-event content, engagement tracking, attendee follow-up.
- **Proposal Review Room**: package summary, value proof, implementation path, legal/security resources, next meeting.
- **Security Review Room**: MSA, infosec, platform governance, deployment model, support path.
- **Team Alignment Room**: role-specific proof for absent stakeholders and a clear internal-review path.
- **AI-To-Activation Room**: AI content creation, Folloze deployment, hosting/governance, personalization, signal capture.

## Quality Gates

Before local preview:

- All visible placeholders are replaced.
- All CTAs do real work.
- Resource cards have real links or in-page content modules.
- No raw meeting-note, Zoom, transcript, or internal task language appears in visible copy.
- No unapproved pricing or commercial terms appear externally.
- The first viewport explains why the buyer is here in less than 10 seconds.
- Each section adds a new reason to act instead of repeating the recap.

Before MCP save:

- Current MCP guide has been read.
- Theme mode is authorized.
- Analytics acknowledgements are true because the HTML actually tracks CTAs and interactions.
- Desktop and mobile preview have been checked.
- Existing-board versus net-new-board intent is explicit.
- Account identity and board name are correct.

## Content Rules

- Use `resources` only for real artifacts or in-page summaries. Do not create dead resource tiles.
- Use `proposal` language only if the user approves it or the source already references a proposal follow-up.
- Use `security` or `MSA` sections only when the buyer asked for them or the deal stage supports them.
- Keep the buyer-facing page concise. Put long internal synthesis in the brief, not the page.
