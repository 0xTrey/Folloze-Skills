## Source Priority

For internal Folloze work:

1. Use current user-provided deal, order-form, pricing, product, or customer context first.
2. Use the internal Google Doc for broad capability definitions and package-row explanation.
3. Use the local markdown source if Drive is unavailable but local repo access exists.
4. Use `product-capabilities-customer-ready.md` for buyer-facing wording.
5. Use `product-capabilities-public-safe.md` for public, repo-safe, or no-Drive contexts.

## Usage Rules

- Treat the Google Doc as internal enablement, not automatically public copy.
- Do not paste internal source notes, Drive provenance, open items, or private data-provider strategy into customer-facing output.
- Do not quote current package entitlements, included user counts, credit quantities, roadmap timing, or connector scope unless the user asks for internal commercial context or provides a current order-form/pricing source.
- If Drive access is unavailable, do not invent current package rows or AI/Data credit burn rules.
- When writing customer-facing material, translate internal caveats into clean qualification language such as `available by package and configuration`.

## Customer-Facing Conversion

When turning internal capability context into customer-facing output:

- Lead with buyer outcomes and workflows.
- Use `Build. Activate. Signal.` when it fits the asset.
- Describe product capabilities in plain language.
- Avoid internal taxonomy unless the buyer already knows the terms.
- Replace internal uncertainty with precise qualification.
- Keep pricing, packaging, roadmap, and credit details tied to a current commercial source.
